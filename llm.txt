# Agent Skills Usage Guide

This repository contains independently loadable skills for tool-using coding agents. Treat the Agent Skills specification, not a particular client, as the common contract.

## Discovery Contract

During discovery, read only `name`, `description`, and path from each `skills/*/SKILL.md`. Load a full skill body only after the request matches its description or the user explicitly invokes it.

Descriptions are trigger contracts. They should state what the skill does and front-load when it applies because large skill lists may be truncated. Workflow details belong in the body. Client UI help belongs only in an optional client adapter.

## Portability Contract

- `skills/<name>/SKILL.md` and its relative resources are the single behavior source of truth.
- Use the standard `name` and `description` fields for discovery. Add standard optional fields only when they carry real portable value.
- Describe needed capabilities rather than vendor tool names, channels, path variables, invocation sigils, or models.
- Use relative paths for bundled resources. Let each client resolve the installed skill root.
- Do not duplicate the workflow for Codex, Claude Code, OpenClaw, Hermes, or individual models.
- Treat `agents/openai.yaml` as optional Codex presentation metadata. Its presence or absence does not change whether the skill is valid for another Agent Skills-compatible client.

## Shared Execution Contract

- Start from the requested outcome, boundaries, downstream consumer, and observable done criteria.
- Inspect repository, runtime, data, and current external evidence before asking the user for facts that tools can resolve.
- Classify uncertainty as a discoverable fact, reversible assumption, consequential decision, or missing authority.
- Make safe project-native assumptions when they do not change the requested result; ask only for material choices.
- Treat the newest explicit user correction as authoritative.
- For time-sensitive claims, record the exact date, environment, version, or data cutoff.
- Separate verified facts, inferences, assumptions, and unresolved proof gaps when the distinction affects the outcome.
- Scale effort and artifacts to risk; do not force every task through a fixed workflow.
- Keep static, code, artifact/read-back, runtime, and live evidence distinct.
- Review-only, diagnosis-only, and planning-only requests do not authorize implementation. End-to-end build requests may move across skills without ceremonial gates when scope and authority remain unchanged.
- Do not present local work as deployed, fresh, migrated, backfilled, published, or live without direct evidence.

## Skill Routing

- `architecture-review`: review existing code for evidence-backed structural drift, hidden ownership, duplicated rules, zombie paths, utility bypass, disproportionate complexity, testability, or operability. Do not implement the refactor.
- `diagnose`: explain a concrete wrong behavior through the real code, runtime, deployment, provider, or data path. Implement a fix only when the request includes fixing or restoring behavior.
- `grill-plan`: converge materially ambiguous goals, scope, success criteria, or consequential product choices after resolving discoverable facts.
- `improve-codebase-architecture`: find deepening opportunities or directly design and compare module interfaces, seam placement, adapters, test surfaces, and AI-navigable code shapes. It does not require a prior architecture-review finding.
- `implement`: carry a clear feature, refactor, test, workflow, UI, API, schema, or documentation task through coherent edits and proportionate verification.
- `task-breakdown`: sequence approved work only when size, risk, dependencies, rollout, or reviewability make one coherent implementation slice unsafe.
- `teach`: run a multi-session learning workspace with a mission, trusted resources, real project, practice, feedback, references, and evidence-based learning records.
- `technical-plan`: design an end-to-end technical contract for mostly settled behavior, including ownership, state, data flow, failures, recovery, observability, migration, rollout, verification, and final boundaries; use `improve-codebase-architecture` instead for focused alternative interfaces or seam placement.
- `to-prd`: write a decision-faithful PRD, implementation brief, or self-contained consultation brief from accepted requirements and approved technical decisions.
- `visual-design-craft`: design, polish, or review web, mobile, or desktop product interfaces through platform-aware states and rendered screenshot evidence.

## Composition

The skills form a routing graph, not a mandatory chain. A task may go directly to the owning skill:

```text
ambiguity -> grill-plan -> technical-plan (only if design choices remain)
failure -> diagnose -> implement (only if fixing was requested)
review -> architecture-review -> improve-codebase-architecture (when design options help)
module/interface/seam design -> improve-codebase-architecture (directly)
approved large work -> task-breakdown -> implement
clear work -> implement
visible UI work -> visual-design-craft, optionally with implement
```

If the user requested end-to-end completion, transition between applicable skills in the same task once no material choice or authority gap remains. If the user requested only a plan, review, explanation, or diagnosis, stop at that outcome.

## Repository Maintenance

- Runtime content lives under `skills/<name>/`.
- `SKILL.md` frontmatter follows the Agent Skills specification and contains only portable metadata.
- Optional client adapters must not redefine triggers, workflow, permissions, or completion criteria from the portable skill.
- Keep reusable skills free of local usernames, machine paths, temporary incidents, current prices, or project-specific names unless the skill is intentionally scoped to that project.
- Preserve progressive disclosure; put conditional detail in directly linked references.
- Use subagents only when the user or applicable instructions explicitly request delegation or parallel agent work.
- After a public trigger or routing change, update the relevant catalog text. Refresh a client adapter only when it exists and becomes stale.
- Prefer `skills-ref validate` for the portable package. Use client-native validation only for the clients actually being tested, then run affected scripts and `git diff --check`.
