# LLM Usage Guide

This repository contains portable workflow skills for Codex-style coding
agents. Treat each directory under `skills/` as an independently loadable skill.

## Discovery

Read only the YAML frontmatter in each `skills/*/SKILL.md` during discovery:

- `name`: canonical skill identifier.
- `description`: when the skill should be used.

Load the full `SKILL.md` body only after the current user request matches that
skill. Do not load every skill body up front.

## Skills

- `architecture-review`: use for architecture reviews, refactoring
  opportunities, module boundaries, seams, and testability.
- `diagnose`: use when the starting point is a bug, regression, flaky behavior,
  production issue, wrong output, or unknown root cause.
- `grill-plan`: use when requirements, user goals, boundaries, rejected
  options, or success criteria need convergence before technical planning, PRD,
  task breakdown, or implementation.
- `implement`: use when the task is already clear enough to implement with a
  narrow edit set and focused verification.
- `task-breakdown`: use after approved requirements, PRD, or technical plan to
  create sequenced implementation tasks with acceptance and verification gates.
- `technical-plan`: use after requirements are clear enough to design
  architecture, module boundaries, contracts, security, observability, rollout,
  and testing strategy.
- `to-prd`: use when converting approved requirements or technical plans into a
  PRD or implementation brief.

## Workflow

For simple tasks, invoke the narrowest matching skill directly. For multi-turn
or high-risk work, prefer:

```text
grill-plan -> technical-plan -> to-prd -> task-breakdown -> implement
```

Skills support inline mode when the current prompt is sufficient and artifact
mode when a checkpoint path is provided. If missing history or compaction would
make the next step unreliable, reconstruct the minimal source contract first or
ask for the missing checkpoint instead of silently relying on chat memory.

## Repository Layout

Runtime skill content lives under `skills/`.

Each `skills/<name>/` directory may contain:

- `SKILL.md`: required runtime instructions for the agent.
- `agents/openai.yaml`: optional UI metadata and default prompt text.
- Optional bundled resources if a future skill needs scripts, references, or
  assets.

This repository is intended to be consumed by the external `skills` CLI, for
example `npx skills@latest add <owner>/<repo>`. It is not itself an npm package.

## Agent Rules

- Prefer `SKILL.md` over README content for runtime behavior.
- Preserve progressive disclosure: read extra files only when the selected
  skill asks for them or the task requires them.
- Do not assume any local machine path, username, shell profile, or global
  install directory. Resolve paths from the current repository and the caller's
  environment.
- For `npx skills` installs, let the `skills` CLI handle copy/link behavior.
- For local development checkouts, symlink skill directories manually if edits
  in this repository should remain the source of truth.
